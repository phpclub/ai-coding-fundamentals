## 1. Проблемы базового RAG

### 🤔 Что не так с "наивным" RAG?

Простой RAG (chunking → embeddings → поиск) хорош для демо, но в production возникают проблемы:

```
┌─────────────────────────────────────────────┐
│    ТИПИЧНЫЕ ПРОБЛЕМЫ NAIVE RAG              │
├─────────────────────────────────────────────┤
│                                             │
│  1️⃣ Пропускает точные совпадения            │
│  ────────────────────────────────           │
│  Ищешь: "iPhone 17 Pro"                     │
│  Находит: статьи про смартфоны вообще       │
│  НЕ находит: точное упоминание модели       │
│                                             │
│  2️⃣ Маленькие chunks без контекста          │
│  ─────────────────────────────────          │
│  Chunk: "...результат составил 42%"         │
│  Проблема: 42% чего? Контекст потерян!      │
│                                             │
│  3️⃣ Неточные запросы пользователей          │
│  ──────────────────────────────────         │
│  Спрашивают: "как настроить вайфай"         │
│  Нужно найти: WiFi, Wi-Fi, беспроводная сеть│
│                                             │
│  4️⃣ Много нерелевантных результатов         │
│  ───────────────────────────────────        │
│  Топ-20 результатов:                        │
│  • 5 релевантных                            │
│  • 15 мусорных                              │
│  LLM читает всё → путается                  │
│                                             │
│  5️⃣ Chunking разрывает логику               │
│  ────────────────────────────────           │
│  "...компания показала рост. В следующем..."│
│         ↑ chunk 1      ↑ chunk 2            │
│  Потеряна связь между chunks!               │
│                                             │
└─────────────────────────────────────────────┘
```

### 📊 Исследования показывают

```
Keywords Studios (2024-2025):
─────────────────────────────
• 45% AI-кода содержит уязвимости безопасности
• Code duplication вырос в 8 раз
• Code churn удвоился

Причина: плохой retrieval в RAG!
```

**Вывод:** Базовый RAG нужно улучшать!

## 2. Гибридный поиск (Hybrid Search)

### 🎯 Идея

```
Объединить два типа поиска:

1. BM25 (keyword search)    ← точные слова
2. Vector search            ← смысл

= Hybrid Search             ← лучшее из обоих
```

### 💡 Аналогия

```
Ищешь друга:

📸 По фотографии (vector)    ← "похож на..."
📝 По имени в книжке (BM25)  ← "точно Иван"

Hybrid = оба способа одновременно!
```

---

### 🔍 BM25: "Дедушка" всех поисковиков

**Что такое BM25?**

```
BM = Best Matching
25 = двадцать пятая версия формулы

История: 1970-е → 1990-е → BM25

Как работает:
─────────────
1. Считает частоту слов в документе
2. Учитывает редкость слов (IDF)
3. Нормализует по длине документа

Формула (упрощённо):
────────────────────
Score = (frequency × IDF) / (frequency + k)

где:
• frequency = сколько раз слово в документе
• IDF = насколько редкое слово
• k = параметр нормализации
```

**Пример:**

```
Запрос: "iPhone 17"

Document 1: "iPhone 17 Pro Max review..."
            ↑      ↑
            3x     2x упоминаний
BM25 score: 15.2

Document 2: "Smartphones and technology..."
            (нет точных слов)
BM25 score: 0.0

Document 3: "iPhone models comparison..."
            ↑
            1x упоминание
BM25 score: 5.1
```

**Плюсы BM25:**
- ✅ Очень быстрый
- ✅ Находит точные совпадения
- ✅ Хорош для имён, кодов, ID

**Минусы BM25:**
- ❌ Не понимает синонимы
- ❌ "iPhone" ≠ "айфон"
- ❌ Чувствителен к опечаткам

---

### 🧠 Vector Search: Семантический поиск

```
Как работает:
─────────────
1. Превращает запрос в embedding
2. Ищет похожие векторы
3. Возвращает по cosine similarity

Пример:
───────
Запрос: "как починить вайфай"
Embedding: [0.2, 0.8, 0.3, ...]

Document 1: "Настройка беспроводной сети"
            [0.19, 0.82, 0.31, ...]
Similarity: 0.95 ← БЛИЗКО!

Document 2: "Python programming guide"
            [-0.5, 0.1, -0.2, ...]
Similarity: 0.12 ← ДАЛЕКО
```

**Плюсы Vector:**
- ✅ Понимает смысл
- ✅ Синонимы работают
- ✅ Устойчив к опечаткам

**Минусы Vector:**
- ❌ Может пропустить точные слова
- ❌ "ID: 12345" найдётся плохо
- ❌ Медленнее BM25

---

### 🔀 Hybrid = BM25 + Vector

```
┌─────────────────────────────────────────┐
│         ГИБРИДНЫЙ ПОИСК                 │
├─────────────────────────────────────────┤
│                                         │
│  Запрос: "iPhone 17 настройка"         │
│     ↓                                   │
│  ┌─────────────────┐                   │
│  │ Параллельно:    │                   │
│  │                 │                   │
│  │  BM25           Vector              │
│  │   ↓              ↓                  │
│  │  Doc A (15.2)   Doc B (0.95)        │
│  │  Doc C (5.1)    Doc A (0.88)        │
│  │  Doc D (2.3)    Doc E (0.82)        │
│  └─────────────────┘                   │
│     ↓                                   │
│  Объединение (RRF)                     │
│     ↓                                   │
│  Финальный рейтинг:                    │
│  1. Doc A (нашли оба метода)           │
│  2. Doc B (высокий vector score)       │
│  3. Doc E (хороший vector)             │
│  4. Doc C (хороший BM25)               │
│  5. Doc D (низкий BM25)                │
│                                         │
└─────────────────────────────────────────┘
```

---

### 🎲 RRF: Reciprocal Rank Fusion

**Проблема:** Как объединить разные шкалы?
- BM25 score: 15.2
- Vector similarity: 0.95

Нельзя просто сложить!

**Решение: RRF**

```
Формула RRF:
────────────
Score = 1/(k + rank)

где:
• rank = позиция в списке
• k = константа (обычно 60)

Пример:
───────
BM25 результаты:
1. Doc A → 1/(60+1) = 0.0164
2. Doc C → 1/(60+2) = 0.0161
3. Doc D → 1/(60+3) = 0.0159

Vector результаты:
1. Doc B → 1/(60+1) = 0.0164
2. Doc A → 1/(60+2) = 0.0161
3. Doc E → 1/(60+3) = 0.0159

Суммируем:
──────────
Doc A: 0.0164 + 0.0161 = 0.0325 ← топ!
Doc B: 0.0164           = 0.0164
Doc C: 0.0161           = 0.0161
Doc E: 0.0159           = 0.0159
Doc D: 0.0159           = 0.0159
```

**Магия RRF:** Фокус на ранге, а не на абсолютных значениях!

---

### 📊 Результаты Hybrid Search

```
Исследования показывают:

Precision улучшается на 30-48%
Recall улучшается на 20-35%

Пример (HotpotQA dataset):
──────────────────────────
Vector only:    45% accuracy
BM25 only:      38% accuracy
Hybrid (RRF):   68% accuracy  ← +50%!
```

## 3. Reranking (Переранжирование)

### 🎯 Идея

```
Проблема: После hybrid search у нас 100 документов,
          но многие всё равно нерелевантны

Решение: Reranker - "умный секретарь"
         который фильтрует лучшие 5-10
```

### 💡 Аналогия

```
Hybrid Search = Помощник принёс 100 писем
                ↓
Reranker      = Секретарь отобрал 5 важных
                ↓
LLM           = Босс читает только важные
```

---

### 🏗️ Архитектура с Reranking

```
┌──────────────────────────────────────────┐
│        ДВУХСТУПЕНЧАТЫЙ RETRIEVAL         │
├──────────────────────────────────────────┤
│                                          │
│  STAGE 1: Broad Retrieval (быстро)       │
│  ─────────────────────────────           │
│  Hybrid Search → 100-200 кандидатов      │
│  Цель: высокий recall (ничего не пропусти│
│                                          │
│        ↓                                 │
│                                          │
│  STAGE 2: Precise Reranking (точно)      │
│  ───────────────────────────────         │
│  Cross-Encoder → топ 5-10 лучших         │
│  Цель: высокий precision (только лучшее) │
│                                          │
│        ↓                                 │
│                                          │
│  В LLM идут только best результаты       │
│                                          │
└──────────────────────────────────────────┘
```

---

### 🔬 Типы Rerankers

#### 1️⃣ Cross-Encoder

```
Как работает:
─────────────
Берёт ПАРУ: (query, document)
Прогоняет через transformer
Выдаёт единый score: 0.0 - 1.0

┌────────────────────────────┐
│   Cross-Encoder Model      │
│                            │
│  Input: [query] [SEP] [doc]│
│          ↓                 │
│    Transformer             │
│          ↓                 │
│    Score: 0.87             │
└────────────────────────────┘

Популярные модели:
──────────────────
• ms-marco-MiniLM-L-6-v2 (compact)
• bge-reranker-large (quality)
• cohere-rerank-v3 (best)

Плюсы:
✅ Очень точный
✅ Глубокое понимание

Минусы:
❌ Медленный (каждая пара отдельно)
❌ Для 100 docs нужно 100 вызовов
```

#### 2️⃣ ColBERT (Late Interaction)

```
Как работает:
─────────────
1. Query → embeddings для каждого токена
2. Doc → embeddings для каждого токена
3. Сравнение на уровне токенов
4. Агрегация в финальный score

Быстрее чем Cross-Encoder!

Пример:
───────
Query: "Python functions"
  Python: [0.2, 0.8, ...]
  functions: [0.5, 0.3, ...]

Document: "How to define functions in Python"
  How: [0.1, 0.2, ...]
  to: [0.3, 0.1, ...]
  define: [0.4, 0.4, ...]
  functions: [0.48, 0.32, ...] ← близко к query!
  in: [0.2, 0.1, ...]
  Python: [0.19, 0.82, ...] ← близко к query!

Score = sum(max similarities)

Плюсы:
✅ Быстрее Cross-Encoder
✅ Точнее обычного vector search
✅ Token-level matching

Минусы:
❌ Сложнее настроить
❌ Больше памяти нужно
```

---

### 📊 Результаты Reranking

```
Улучшение качества:

Pinecone research:
──────────────────
Vector only:        NDCG@10 = 0.52
+ Hybrid:           NDCG@10 = 0.68 (+30%)
+ Hybrid + Rerank:  NDCG@10 = 0.82 (+57%)!

ZeroEntropy (2026):
───────────────────
Baseline retriever: 100% baseline
+ zerank-1 reranker: +28% NDCG@10

Результат: Галлюцинации снижаются на 70-90%!
```

---

### ⚡ Trade-off: Скорость vs Качество

```
┌────────────────────────────────────────┐
│      ВЫБОР СТРАТЕГИИ RERANKING         │
├────────────────────────────────────────┤
│                                        │
│  Без reranking                         │
│  Скорость: 50ms                        │
│  Качество: ⭐⭐⭐                        │
│  Когда: MVP, demo                      │
│                                        │
│  ColBERT reranking                     │
│  Скорость: 200ms                       │
│  Качество: ⭐⭐⭐⭐                      │
│  Когда: Production с балансом          │
│                                        │
│  Cross-Encoder reranking               │
│  Скорость: 2000ms                      │
│  Качество: ⭐⭐⭐⭐⭐                    │
│  Когда: Критичные задачи (legal, med) │
│                                        │
└────────────────────────────────────────┘

Золотое правило:
────────────────
Stage 1 (fast): Hybrid → 100-200 docs
Stage 2 (slow): Rerank → top 5-10
```

## 4. Parent-Child Documents

### 🎯 Проблема

```
Маленькие chunks теряют контекст:

Chunk: "...результат составил 42%"

42% чего?
Когда?
В каком контексте?

Информации не хватает!
```

### 💡 Решение

```
Parent-Child подход:

1. Ищем по маленьким chunks (быстро, точно)
2. Возвращаем большой parent document (контекст)
```

---

### 🏗️ Архитектура

```
┌──────────────────────────────────────────┐
│         PARENT-CHILD СТРУКТУРА           │
├──────────────────────────────────────────┤
│                                          │
│  PARENT DOCUMENT (полная секция)         │
│  ┌────────────────────────────────────┐  │
│  │                                    │  │
│  │  "Q2 2025 Financial Results        │  │
│  │                                    │  │
│  │  Revenue increased by 42%          │  │
│  │  compared to Q2 2024 ($2.3M)       │  │
│  │                                    │  │
│  │  Key factors:                      │  │
│  │  - New product launch              │  │
│  │  - Market expansion                │  │
│  │  - Cost optimization               │  │
│  │                                    │  │
│  │  Forecast: continued growth..."    │  │
│  │                                    │  │
│  └────────────────────────────────────┘  │
│         ↓ разбивается на ↓               │
│                                          │
│  CHILD CHUNKS (для поиска):              │
│  ┌──────────────────────┐                │
│  │ Chunk 1:             │                │
│  │ "Revenue increased   │ ← searchable   │
│  │  by 42%..."          │                │
│  └──────────────────────┘                │
│  ┌──────────────────────┐                │
│  │ Chunk 2:             │                │
│  │ "Key factors:        │ ← searchable   │
│  │  New product..."     │                │
│  └──────────────────────┘                │
│  ┌──────────────────────┐                │
│  │ Chunk 3:             │                │
│  │ "Forecast:           │ ← searchable   │
│  │  continued growth"   │                │
│  └──────────────────────┘                │
│                                          │
└──────────────────────────────────────────┘

Как работает:
─────────────
Запрос: "Рост выручки?"
  ↓
Находит: Chunk 1 (про 42%)
  ↓
Возвращает: ВЕСЬ parent document
  ↓
LLM видит полный контекст!
```

---

### 📊 Результаты

```
Parent-Child vs Simple Chunks:

Контекстная точность: +40%
Полнота ответов: +60%
Снижение "я не знаю": 70%

Пример:
───────
Simple: "Выручка выросла на 42%"
        [непонятно в каком контексте]

Parent-Child: "Q2 2025: выручка $2.3M (+42% YoY)
               за счёт нового продукта и 
               оптимизации затрат"
               [полная картина!]
```

## 5. Query Expansion

### 🎯 Проблема

```
Пользователи формулируют запросы плохо:

Спрашивают: "настроить вайфай"
Нужно найти: WiFi, Wi-Fi, wireless, 
             беспроводная сеть, WLAN
```

### 💡 Решение

```
LLM автоматически создаёт несколько 
версий запроса и ищет по всем!
```

---

### 🔄 Методы Query Expansion

#### 1️⃣ Multi-Query Generation

```
Оригинальный запрос:
"как настроить WiFi"

LLM генерирует варианты:
────────────────────────
1. "конфигурация беспроводной сети"
2. "установка Wi-Fi соединения"
3. "подключение к интернету через роутер"
4. "настройка WLAN"
5. "WiFi setup guide"

Ищем по всем 5 вариантам параллельно
  ↓
Объединяем результаты (RRF)
  ↓
Топ-10 лучших документов
```

#### 2️⃣ HyDE (Hypothetical Document Embeddings)

```
Идея: Пусть LLM создаст гипотетический 
      идеальный ответ, и ищем похожие docs

Запрос: "Как работает RAG?"
  ↓
LLM генерирует гипотетический ответ:
"RAG - это техника которая объединяет
retrieval и generation. Сначала система
ищет релевантные документы..."
  ↓
Превращаем этот ответ в embedding
  ↓
Ищем документы похожие на гипотетический ответ
  ↓
Находим реальные статьи про RAG!
```

#### 3️⃣ Query Decomposition

```
Сложный запрос разбивается на под-вопросы:

Запрос: "Сравни производительность 
         iPhone 17 и Samsung S25 при съёмке"

LLM разбивает:
──────────────
1. "характеристики камеры iPhone 17"
2. "характеристики камеры Samsung S25"
3. "тесты съёмки в слабом освещении iPhone 17"
4. "тесты съёмки в слабом освещении Samsung S25"

Ищем по каждому под-вопросу
  ↓
Собираем всю информацию
  ↓
LLM делает сравнение на основе всех данных
```

---

### 📊 Результаты

```
Multi-Query expansion:
─────────────────────
Recall: +25-40%
Coverage: +50%

Особенно эффективно для:
• Многоязычных запросов
• Запросов с опечатками
• Неточных формулировок
• Сложных вопросов
```

## 6. Semantic Chunking

### 🎯 Проблема "тупого" chunking

```
Fixed-size chunking:

"Компания показала рост в 2025 году.
Выручка составила 42 миллиона долл|
──────────────────────────────────────
chunk 1 ↑                    chunk 2 ↓
──────────────────────────────────────
аров. В следующем квартале ожидается
дальнейшее увеличение показателей."

Проблема:
❌ Разорвано предложение
❌ Потерян контекст
❌ Chunk 2 начинается с "аров"
```

---

### 💡 Semantic Chunking

```
AI анализирует СМЫСЛ текста и разбивает
по логическим границам:

Документ:
─────────
[Раздел 1: История Python]
  Параграф 1.1: Создание языка
  Параграф 1.2: Развитие
  
[Раздел 2: Синтаксис Python]
  Параграф 2.1: Переменные
  Параграф 2.2: Функции

[Раздел 3: Библиотеки]
  ...

Semantic chunking:
──────────────────
Chunk 1: ВСЁ про историю (логически целое)
Chunk 2: ВСЁ про синтаксис (логически целое)
Chunk 3: ВСЁ про библиотеки (логически целое)

Каждый chunk = законченная мысль!
```

---

### 🔬 Методы Semantic Chunking

#### 1️⃣ Sentence Similarity

```
Алгоритм:
─────────
1. Разбиваем на предложения
2. Создаём embeddings для каждого
3. Считаем similarity между соседними
4. Где similarity падает → граница chunk

Пример:
───────
Sentence 1: "Python был создан в 1991"
Sentence 2: "Гвидо ван Россум разработал"
Similarity: 0.92 ← высокая, одна тема

Sentence 3: "Переменные в Python динамические"
Similarity: 0.45 ← низкая, новая тема!

Граница chunk между Sentence 2 и 3
```

#### 2️⃣ LLM-Based Chunking

```
Prompt для LLM:
───────────────
"Проанализируй текст и разбей на 
логические секции. Каждая секция 
должна покрывать одну тему."

LLM анализирует и возвращает:
────────────────────────────
Chunk 1: [строки 1-45]   "История Python"
Chunk 2: [строки 46-120] "Синтаксис"
Chunk 3: [строки 121-200] "Библиотеки"
```

---

### 📊 Результаты

```
Semantic vs Fixed-size:

Контекстная целостность: +70%
Качество ответов: +40%
Снижение "нерелевантного шума": 60%

Но:
─────
Скорость: медленнее в 3-5 раз
Стоимость: +$0.01-0.05 на документ
```

**Рекомендация:** Используй для важных документов (policies, legal, technical specs)

## 7. Contextual RAG

### 🎯 Идея (от Anthropic, 2024)

```
Проблема: Chunks без контекста

Chunk: "За квартал рост 42%"

Вопросы:
• Какой квартал?
• Рост чего?
• Какая компания?

Решение: ДОБАВИТЬ КОНТЕКСТ К CHUNK!
```

---

### 🏗️ Contextual Retrieval

```
┌────────────────────────────────────────┐
│      CONTEXTUAL CHUNKING               │
├────────────────────────────────────────┤
│                                        │
│  Оригинальный документ:                │
│  "Acme Corp Q2 2025 Financial Report   │
│   Revenue increased by 42%             │
│   compared to Q2 2024..."              │
│                                        │
│         ↓ LLM добавляет контекст       │
│                                        │
│  Contextual Chunk:                     │
│  "Документ: Acme Corp Q2 2025 Report   │
│   Секция: Revenue                      │
│   Контекст: Финансовые результаты      │
│   Содержание: Revenue increased by 42% │
│   compared to Q2 2024..."              │
│                                        │
│  Теперь chunk = самодостаточный!       │
│                                        │
└────────────────────────────────────────┘
```

---

### 🔄 Процесс

```
ШАГ 1: Chunking
───────────────
Document → Chunks

ШАГ 2: Context Generation
─────────────────────────
Для каждого chunk:
  
  Prompt для LLM:
  "Документ: [название]
   Chunk: [содержимое]
   
   Создай краткий контекст (2-3 предложения)
   который объясняет о чём этот chunk."
   
  LLM генерирует:
  "Это отрывок из финансового отчёта
   Acme Corp за Q2 2025, раздел Revenue,
   описывающий рост выручки."

ШАГ 3: Prepend Context
──────────────────────
Новый chunk = Context + Original chunk

ШАГ 4: Embedding
────────────────
Создаём embedding для нового chunk
```
