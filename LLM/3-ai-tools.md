## 1.1 Chat-based инструменты

**Примеры:** ChatGPT, Claude.ai, Gemini

### Как работает

```
Ты ← → Браузер ← → Chat AI

1. Копируешь код в чат
2. AI анализирует
3. AI генерирует ответ
4. Копируешь обратно в редактор
```

### Плюсы и минусы

**✅ Плюсы:**
- Простой вход (не нужна установка)
- Хорошо для обучения и объяснения концепций
- Дешёвый/бесплатный
- Можно использовать с мобильного

**❌ Минусы:**
- Нет доступа к твоему проекту
- Копирование туда-сюда
- Нет интеграции с редактором
- Нет выполнения кода
- Контекст теряется между чатами

### Когда использовать

| Задача | Подходит? |
|--------|-----------|
| Изучение нового языка | ✅ Отлично |
| Вопросы по концепциям | ✅ Отлично |
| Разбор ошибок | ✅ Хорошо |
| Генерация сниппетов | ✅ Хорошо |
| Работа над проектом | ❌ Неудобно |
| Рефакторинг | ❌ Неудобно |

### Стоимость (2026)

| Сервис | Free | Pro | Max/Team |
|--------|------|-----|----------|
| ChatGPT | ✅ Базовый | $20/мес | $200/мес |
| Claude | ✅ Базовый | $20/мес | $30/юзер |

## 1.4 Агрегаторы — OpenRouter

### Что такое OpenRouter?

**OpenRouter** = единый API для доступа ко ВСЕМ моделям

```
OpenRouter
    ↓
┌─────────────────────────────────────┐
│ Anthropic (Claude)                  │
│ OpenAI (GPT)                        │
│ Google (Gemini)                     │
│ Meta (Llama)                        │
│ Mistral                             │
│ DeepSeek                            │
│ Qwen                                │
│ ... и ещё 200+ моделей              │
└─────────────────────────────────────┘
```

### Зачем это нужно?

**Проблема:**
```
❌ Без OpenRouter:
- Регистрация в Anthropic → API key
- Регистрация в OpenAI → API key
- Регистрация в Google → API key
- Разные форматы API
- Разная оплата
```

**Решение:**
```
✅ С OpenRouter:
- Одна регистрация
- Один API key
- Единый формат API (OpenAI-compatible)
- Одна оплата
```

### Использование с Continue

```json
// ~/.continue/config.json
{
  "models": [
    {
      "title": "Claude Sonnet 4.5 (OpenRouter)",
      "provider": "openrouter",
      "model": "anthropic/claude-sonnet-4.5",
      "apiKey": "YOUR_OPENROUTER_KEY"
    },
    {
      "title": "GPT-5 (OpenRouter)",
      "provider": "openrouter",
      "model": "openai/gpt-5",
      "apiKey": "YOUR_OPENROUTER_KEY"
    },
    {
      "title": "DeepSeek-V3 (OpenRouter)",
      "provider": "openrouter",
      "model": "deepseek/deepseek-v3",
      "apiKey": "YOUR_OPENROUTER_KEY"
    }
  ]
}
```

### Преимущества OpenRouter

**1. Сравнение моделей:**
```
Одна задача → тестируешь на 5 моделях → выбираешь лучшую
```

**2. Fallback (запасной вариант):**
```
Claude недоступен → автоматически переключается на GPT
```

**3. Оптимизация стоимости:**
```
Простые задачи → дешёвая модель (Qwen)
Сложные задачи → дорогая модель (Claude Opus)
```

## 2. РЕЖИМЫ РАБОТЫ: PLANNING VS EXECUTE

### 🔥 Критически важная концепция!

**Проблема "бездумного выполнения":**

```
❌ ПЛОХО: Режим Execute сразу

Ты: "Перепиши это на TypeScript"

AI сразу начинает писать код:
├─ Переписывает auth.js → auth.ts
├─ Забыл про типы в utils
├─ Забыл обновить импорты
├─ Забыл про тесты
└─ Результат: Код работает, но архитектура сломана
```

```
✅ ХОРОШО: Сначала Planning, потом Execute

Ты: "Перепиши это на TypeScript. 
     Сначала составь план."

AI планирует:
┌─────────────────────────────────────────┐
│ ПЛАН РЕФАКТОРИНГА:                      │
│                                         │
│ 1. Анализ зависимостей                  │
│    - auth.js импортирует utils.js       │
│    - Header.jsx использует auth         │
│                                         │
│ 2. Очерёдность конвертации              │
│    a) utils.js → utils.ts (базовый)     │
│    b) auth.js → auth.ts (зависит от a)  │
│    c) Header.jsx → Header.tsx           │
│                                         │
│ 3. Типы                                 │
│    - Создать types/auth.ts              │
│    - Создать types/user.ts              │
│                                         │
│ 4. Тесты                                │
│    - Обновить auth.test.js → .ts        │
│                                         │
│ 5. Конфигурация                         │
│    - Обновить tsconfig.json             │
│    - Обновить package.json              │
└─────────────────────────────────────────┘

Ты проверяешь план → соглашаешься

AI выполняет план шаг за шагом ✅
```

---

### Типы режимов в AI инструментах

#### **1. Planning Mode (режим планирования)**

**Что делает:**
- Анализирует задачу
- Составляет план
- Показывает план тебе
- Ждёт подтверждения
- Только потом выполняет

**Когда использовать:**
- ✅ Рефакторинг
- ✅ Архитектурные изменения
- ✅ Миграции (Redux → Zustand, JS → TS)
- ✅ Большие фичи
- ✅ Когда не уверен в подходе

**Примеры команд:**

```bash
# Claude Code
claude-code --mode plan "Мигрируй на TypeScript"

# Cursor
Ctrl+K → "Plan: Add authentication"

# В чате
"Сначала составь план, как добавить аутентификацию"
```

---

#### **2. Execute Mode (режим выполнения)**

**Что делает:**
- Сразу начинает писать код
- Не показывает план
- Быстро, но рискованно

**Когда использовать:**
- ✅ Простые задачи
- ✅ Небольшие фиксы

## 3. PROMPT ENGINEERING ДЛЯ VIBE-CODING

### 3.1 Основы хорошего промпта

**Анти-паттерн (так делать НЕ надо):**

```
❌ "Сделай калькулятор"

Проблема:
- Слишком расплывчато
- Непонятен стек
- Нет требований к дизайну
- Нет указания на функции
```

**Правильный промпт:**

```
✅ "Создай веб-калькулятор:

СТЕК:
- React + TypeScript
- Tailwind CSS

ФУНКЦИИ:
- Операции: +, -, *, /, %
- Очистка (C)
- Удаление последнего символа (←)
- История последних 5 операций

ДИЗАЙН:
- Минималистичный
- Крупные кнопки (touch-friendly)
- Тёмная тема
- Responsive (мобильный first)

ТРЕБОВАНИЯ:
- Валидация ввода
- Обработка ошибок (деление на 0)
- Keyboard shortcuts
- Тесты на основные функции"
```

### 3.2 Фреймворк: Role → Goal → Constraints

**Структура идеального промпта:**

```
┌─────────────────────────────────────────┐
│ ПРОМПТ = Role + Goal + Constraints      │
├─────────────────────────────────────────┤
│                                         │
│ 1. ROLE (Роль)                          │
│    Кто ты, какая экспертиза             │
│                                         │
│ 2. GOAL (Цель)                          │
│    Что нужно сделать                    │
│                                         │
│ 3. CONSTRAINTS (Ограничения)            │
│    Как делать, что использовать         │
└─────────────────────────────────────────┘
```

**Пример:**

```
═══════════════════════════════════════════
ROLE (Роль):
───────────────────────────────────────────
Ты senior React разработчик со знанием:
- TypeScript
- Accessibility (WCAG 2.1)
- Tailwind CSS
- React Hook Form
═══════════════════════════════════════════
GOAL (Цель):
───────────────────────────────────────────
Создай переиспользуемый компонент Button
с поддержкой разных вариантов и размеров
═══════════════════════════════════════════
CONSTRAINTS (Ограничения):
───────────────────────────────────────────
ТЕХНОЛОГИИ:
- TypeScript с strict mode
- Tailwind CSS (без кастомного CSS)
- Compound components pattern

ВАРИАНТЫ:
- primary, secondary, outline, ghost
- 3 размера: sm, md, lg

ACCESSIBILITY:
- Правильные ARIA атрибуты
- Keyboard navigation
- Screen reader friendly

ТЁМНАЯ ТЕМА:
- Автоматическая поддержка dark mode

ЭКСПОРТ:
- Named export
- Типы экспортируются отдельно
```

### 3.7 Интерактивное задание

**Задание:** Напиши промпт по фреймворку для реальной задачи

**Цель:** Создать компонент поиска с автодополнением

**Твой промпт должен включать:**
1. ✅ Role (роль)
2. ✅ Goal (цель)
3. ✅ Constraints (ограничения)
4. ✅ Behavior (поведение, не implementation)

**Пример правильного решения:**

```
═══════════════════════════════════════════
ROLE:
───────────────────────────────────────────
Ты React/TypeScript разработчик 
со знанием accessibility и UX

═══════════════════════════════════════════
GOAL:
───────────────────────────────────────────
Создай компонент SearchBar с автодополнением

═══════════════════════════════════════════
CONSTRAINTS:
───────────────────────────────────────────
СТЕК:
- TypeScript strict mode
- Headless UI (не Tailwind UI)
- Tailwind CSS

ФУНКЦИИ:
- Debounce 300ms
- Минимум 3 символа для поиска
- До 5 результатов в dropdown
- Highlight совпадений

ACCESSIBILITY:
- ARIA attributes
- Keyboard navigation (↑↓, Enter, Esc)
- Screen reader friendly
- Focus management

UX:
- Spinner при загрузке
- Empty state при отсутствии результатов
- Esc закрывает dropdown
- Клик вне закрывает dropdown
- Автофокус при монтировании

═══════════════════════════════════════════
BEHAVIOR (не implementation!):
───────────────────────────────────────────
При вводе текста:
- Показать loading indicator
- Отправить запрос к API (debounced)
- Показать результаты в dropdown

При выборе результата:
- Заполнить input
- Закрыть dropdown
- Emit onChange event

При ошибке:
```
